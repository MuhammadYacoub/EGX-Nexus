# EGX-Nexus: هيكلية النظام، الوضع الحالي، خارطة الطريق، ودليل الذكاء الاصطناعي (AI Prompt & Roadmap)

**الإصدار:** 1.0
**الهدف:** توثيق شامل للوضع الحالي لمنصة EGX-Nexus، وتوضيح الرؤية المستقبلية لتطويرها، مع توفير دليل شامل (Prompt/Context) لوكلاء الذكاء الاصطناعي (AI Agents) الذين سيعملون على بناء وتوسعة المشروع في المستقبل.

---

## الجزء الأول: الوضع الحالي للنظام (Current State Architecture)

بناءً على تحليل الكود البرمجي الحالي للمنصة (قواعد البيانات والخدمات المصغرة)، هذه هي حالة النظام الأساسية:

### 1. مصادر البيانات والتخزين (Data Store & Databases)
النظام يعتمد على بنية تحتية هجينة وموزعة للتخزين:
*   **MySQL (البيانات الهيكلية والتحليلية التاريخية):**
    *   `stocks`: بيانات الأسهم الأساسية (الرمز، الاسم، القطاع).
    *   `stock_prices`: البيانات التاريخية اليومية OHLCV (الفتح، الإغلاق، الأعلى، الأدنى، الحجم).
    *   `stock_features`: ميزات التحليل الفني والإشارات المولدة (`rsi`, `macd` وغيرها، بالإضافة للإشارة `signal`).
    *   *ملاحظة:* **لا يوجد حالياً أي جدول للمستخدمين أو المحافظ المالية.**
*   **TimescaleDB / PostgreSQL (البيانات اللحظية والزمنية):**
    *   `order_book_l1`: بيانات سجل الأوامر اللحظية من المستوى الأول (أفضل `bid` وأفضل `ask` مع أحجامها).
    *   `institutional_activity`: نشاط المؤسسات ورصد الأحجام غير المعتادة (`anomaly_score`).
*   **Redis (ناقل الرسائل والذاكرة المؤقتة):**
    *   العمود الفقري (Message Broker) للتواصل بين الخدمات.
    *   بث الإشارات اللحظية (`Pub/Sub`) من `prediction-engine`.
    *   تخزين مؤقت (Caching) لجلسات الاتصال (Sessions) وأوقات جمع البيانات (`last_harvest`).

### 2. الخدمات المصغرة الحالية (Microservices)
*   **`auth-gateway` (بوابة المصادقة):**
    *   **دورها الحالي:** مخصصة **لدخول النظام إلى مصادر البيانات** (مثل الوسطاء: Mubasher, Thndr) وليس لتسجيل دخول المستخدمين العاديين للمنصة.
    *   تقوم بتوليد جلسات (Sessions) وتخزينها في Redis ليستخدمها النظام في جلب البيانات اللحظية.
*   **`prediction-engine` (محرك التنبؤ والبيانات اللحظية):**
    *   تعمل كـ Listener متصل بالوسطاء الماليين (عبر WebSocket مثل TradingView Protocol).
    *   تلتقط البيانات اللحظية (QuoteTicks)، وتقوم بتوحيدها (Normalization)، ثم تبثها عبر Redis ليتم استهلاكها من الخدمات الأخرى.
*   **`data-harvester` (جامع البيانات):**
    *   خدمة تعتمد على Cron Jobs (يومياً بعد الإغلاق) لجلب البيانات التاريخية OHLCV من مصادر مثل Yahoo Finance وتحديث قاعدة البيانات `MySQL`.
*   **`training-pipeline` (نظام التدريب الآلي):**
    *   خدمة مستقلة مبنية بـ Python، تستورد بيانات `stock_features` التاريخية وتدرب نموذج تعلم آلي تقليدي (`Gradient Boosting Classifier`) لتوقع الإشارات (شراء، بيع، احتفاظ)، وتحفظ النموذج في `models/`.
*   **`dashboard-api` و `notification-hub`:**
    *   **حالتهما:** حالياً عبارة عن قوالب فارغة (Skeletons) تحتوي فقط على مسارات فحص الصحة (`/health`).
    *   `notification-hub` مهيأة بنيوياً للاتصال بتيليجرام (عبر متغيرات البيئة) لكنها تفتقر لمسارات إرسال واستعلام الإشعارات المكتملة.
    *   `dashboard-api` لا تحتوي على مسارات للاستعلام عن البيانات حالياً.

## الجزء الثاني: خارطة الطريق والرؤية المستقبلية (Roadmap & Proposed Architecture)

للانتقال بالنظام من أداة شخصية/تحليلية بحتة إلى منصة متكاملة وذكية، يجب تطوير الخدمات التالية:

### 1. نظام إدارة المستخدمين والمحفظة الإرشادية (Users & Model Portfolio)
*   **المحفظة الإرشادية (Model Portfolio):** محفظة مركزية تديرها قرارات الذكاء الاصطناعي الخاص بالمنصة.
*   **تعدد المستخدمين:** توسعة قاعدة بيانات `MySQL` لتشمل جداول `users`, `user_preferences`, و `user_portfolios`.
*   **حرية النسخ (Copy Trading):** يمكن للمستخدمين فقط مراقبة أداء "المحفظة الإرشادية" أو تفعيل خيار مطابقة صفقاتهم (بشكل آلي أو يدوي عبر إشعارات) مع قرارات النظام.
*   **توسعة `auth-gateway`:** لدعم تسجيل الدخول الفعلي للمستخدمين (JWT Auth) بالإضافة لدورها الحالي مع مصادر البيانات.

### 2. نظام الاستعلام الشامل (Advanced Query API - Dashboard)
*   تطوير خدمة `dashboard-api` لتكون الواجهة الخلفية الكاملة (Backend for Frontend).
*   **مسارات مطلوبة (Routes):**
    *   استعلام عن الأسهم وحالتها الحالية والمؤشرات الفنية.
    *   سجل الإشارات (Signal History) ونسب النجاح.
    *   تتبع أداء "المحفظة الإرشادية" (الأرباح، الخسائر، الصفقات المفتوحة).
    *   إدارة حساب المستخدم وتفضيلات الإشعارات.

### 3. مستقبل الذكاء الاصطناعي في المنصة (AI Core Evolution)
النظام سيعتمد بنية هجينة ومتعددة الوكلاء (Hybrid Multi-Agent Architecture):
*   **التعلم العميق (Deep Learning Engine):** دمج نماذج متقدمة (LSTMs أو Transformers) لتحليل التسلسل الزمني للأسعار بطريقة أعقد من `training-pipeline` الحالي.
*   **تحليل المشاعر (Ollama NLP):** تشغيل وكيل مخصص مدعوم بـ Ollama محلياً لتحليل الأخبار الاقتصادية باللغة العربية وتأثيرها على معنويات السوق (Sentiment Analysis).
*   **الوكيل التفاوضي (Agent Negotiation in `core-brain`):** جعل خدمة `core-brain` تعمل كمنسق أو "قاضٍ" يستقبل التوصيات من (التحليل الفني في `analysis-engine`، الـ Machine Learning التقليدي، التعلم العميق، وتحليل الأخبار)، ويقوم بالجمع بينها لإصدار القرار النهائي للمحفظة الإرشادية.

### 4. تطوير الـ Notification Hub
*   تحويله من نظام بث (Push) فقط إلى نظام متكامل.
*   تخزين الإشعارات في قاعدة البيانات لدعم الاستعلام (History) عبر `dashboard-api`.
*   دعم تخصيص الإشعارات (تنبيهات لأسهم معينة يختارها المستخدم).

## الجزء الثالث: دليل و إرشادات وكلاء الذكاء الاصطناعي (AI Agents Prompt & Guidelines)

هذا القسم موجه لأي وكيل ذكاء اصطناعي (AI Agent) يقوم بقراءة هذا الملف لفهم سياق المشروع والمساعدة في كتابة الكود:

### تعليمات عامة للوكلاء (Core AI Directives)
1.  **المصطلحات الحيادية:** عند كتابة الكود أو التوثيق، لا تستخدم أسماء منصات أو وسطاء محددين (مثل Thndr أو Mubasher) في التسميات العامة أو الـ UI. استخدم مصطلحات عامة مثل `Local Broker API`، `Data Source`، أو `Broker Account`.
2.  **البنية التحتية والبيئة (Local-First):** النظام مصمم ليكون مستقلاً ويعمل محلياً بالكامل (Local-First AI).
    *   **ممنوع استخدام LLMs خارجية (مثل OpenAI API) لتحليل السوق.**
    *   تحليل النصوص والأخبار يتم حصرياً عبر `Ollama` المحلي (بمنفذ 11434).
    *   التحليل المالي يعتمد على مكتبات رياضية دقيقة (`technicalindicators`) أو نماذج مدربة داخلياً (ML/DL).
3.  **هيكلة الخدمات المصغرة (Microservices):**
    *   كل خدمة جديدة يجب أن توضع في مجلدها الخاص تحت `services/`.
    *   تطبيقات Node.js يجب أن تستخدم نظام وحدات ES (`"type": "module"`).
    *   يجب ألا تتصل الخدمات بقواعد البيانات بشكل مباشر إذا كان هناك خدمة وسيطة مخصصة لذلك (مثلاً البيانات تمر عبر Redis أو APIs).
4.  **التزامن وقواعد البيانات (Database Integrity):**
    *   الاعتماد على إطار العمل الخاص بـ `ON DUPLICATE KEY UPDATE` أو ما يوازيه عند إدخال البيانات لتجنب تكرار البيانات الزمنية.
    *   يجب أن تكون الجداول قابلة للتوسع. أي تعديل على `MySQL` أو `TimescaleDB` يجب أن يكون مصحوباً بملفات التهيئة الأساسية `init.sql`.
5.  **الرد السلس والآمن (Graceful Fallbacks):**
    *   في حال فشل الاتصال بـ `Redis` أو أي قاعدة بيانات أخرى، يجب على الخدمة تسجيل الخطأ (Log Warning) والاستمرار في تقديم ما يمكن تقديمه عبر الـ HTTP من دون التوقف الكامل للتطبيق (Crash).
6.  **نمط رسائل الـ Commit:**
    *   يجب أن تتبع الـ Commits النمط القياسي: `type(scope): description`. (مثال: `feat(dashboard-api): add portfolio performance routes`).

### تعليمات خاصة بالتطوير القادم (Next Tasks Context)
*   **عند طلب إنشاء نظام المستخدمين:** قم بتعديل `services/data-store/mysql/init.sql` لإضافة الجداول (`users`, `user_portfolios`, `model_portfolio_trades`)، ثم ابدأ بتطوير `dashboard-api` للتعامل معها.
*   **عند طلب إنشاء نظام الاستعلام:** ركز على `dashboard-api`، وقم بربطه بـ `MySQL` و `TimescaleDB` لتوفير `GET` endpoints للبيانات التاريخية واللحظية وأداء المحفظة.
*   **عند العمل على إشعارات المستخدمين:** قم بتوسيع `notification-hub` ليتضمن مسارات لتسجيل تنبيهات مخصصة (Custom Alerts) للمستخدمين وربطها بـ Telegram ID الخاص بكل مستخدم إن وجد.
